Как с нами работать — 3 шага
ШАГ 01
Выберите тип задачи и заполните форму в Jira
Описание, ожидаемый результат, эталон данных. Чем точнее формулировка — тем меньше итераций.
ШАГ 02
Дождитесь ответа PM с оценкой и сроком
PM проанализирует запрос, согласует уточнения и зафиксирует срок выполнения в задаче Jira.
ШАГ 03
Будьте на связи на этапах теста и UAT
Своевременные ответы и согласования напрямую влияют на скорость публикации в Production.
Важно: запросы вне Jira (мессенджеры, e-mail, устные просьбы) в работу не принимаются. Это правило защищает обе стороны: заказчик получает прозрачную историю задачи, команда — корректную приоритизацию.

Статус «Невозможно удовлетворить / Отмена» доступен на любом этапе — для всех четырёх типов заявок. Заявка переводится в этот статус с комментарием о причине; после корректировки заказчик возвращает её в статус «В очередь» либо закрывает окончательно.
🧭 Правила выбора типа задачи
Создание нового приложения с нуля → всегда «Проект», независимо от объёма.
Изменение программного кода (скрипты, выражения, формулы) → «Доработка», даже если задача начиналась как визуальная.
Только визуальное оформление без правок логики → «Дизайн».
Учётки, лицензии, доступы, рассылки, документация, обучение → «Администрирование».

Ориентировочные сроки по этапам
Диапазоны min/max, основанные на типовой практике. Окончательный срок фиксируется в задаче Jira после оценки.
⚠️
Сроки являются ориентировочными, а не обязательством Указанные диапазоны отражают типовую длительность этапов для задач средней сложности. Реальный срок зависит от объёма работ, сложности модели данных, доступности заказчика на этапах согласования и тестирования, а также параллельной загрузки команды. Конкретные сроки по каждой задаче определяются после анализа и фиксируются в задаче Jira; именно они являются согласованным сроком исполнения. Сроки на сайте справочные и не могут служить основанием для претензий.

Роли и зоны ответственности
Кто за что отвечает в текущей структуре отдела и при взаимодействии с бизнесом.
🎯
Руководитель проекта (PM)
Единая точка коммуникации между бизнесом и командой разработки
  • Принимает и валидирует входящие запросы от бизнеса
  • Назначает приоритет задачам внутри команды
  • Распределяет работу между разработчиками с учётом их загрузки
  • Контролирует ход выполнения и соблюдение этапов workflow
  • Является единой точкой коммуникации с бизнес-заказчиком
💻
Центральная команда разработчиков
Разработка, code review, поддержка среды
  • Выполняет анализ требований, разработку и доработку приложений
  • Проводит самотестирование перед передачей на code review
  • Проводит code review задач коллег по команде
  • Поддерживает централизованную среду Qlik Sense на всех объектах
  • Ежедневно обновляет статус задач в Jira, своевременно сигнализирует о блокерах
  • Проводит мониторинг после публикации в Production
📍
Локальные специалисты на объектах
Связующее звено между бизнесом объекта и Центральной командой
  • Помогает бизнесу объекта корректно сформулировать запрос в Jira
  • Уточняет специфику данных и процессов на стороне объекта
  • Координирует тестирование на стороне объекта
  • Не выполняет самостоятельную разработку и не вносит изменения в среду
  • Сопровождает выданные на объект лицензии в части пользователей
🏢
Бизнес-заказчик
Подразделения объектов холдинга AKFA
  • Заполняет форму приёма задачи в Jira полностью и достоверно
  • Предоставляет эталонные данные для сверки и валидации
  • Отвечает на уточняющие вопросы команды в разумный срок
  • Проводит тестирование и UAT в согласованные сроки
  • Согласовывает приёмку результата перед публикацией в Production
  • Не обращается напрямую к разработчикам в обход PM и Jira
🗄
Владелец источника данных
Подключается, когда задача затрагивает данные из ERP/DB
  • Подтверждает корректность модели данных и используемых таблиц
  • Согласовывает доступ к источнику и характер извлекаемых данных
  • Отвечает на вопросы по бизнес-логике источника
  • Подключается к UAT при существенных изменениях модели
